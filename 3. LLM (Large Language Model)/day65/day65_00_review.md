# Day65 Review: 3차 단위 프로젝트 한자 데이터 정제 및 운영 후보군 검증

## 1. 개요

이 문서는 SKN29 3차 단위 프로젝트에서 2026년 6월 16일에 진행한 한자 데이터 정제 및 운영 후보군 검증 작업을 정리한 기록이다.

작업의 핵심은 한자 작명 QA 시스템에서 사용할 한자 데이터를 운영 가능한 수준으로 정제하고, 전체 후보군 중 어떤 범위를 ChromaDB 운영 기본값으로 사용할지 판단하는 것이었다.

단순히 한자를 많이 넣는 것이 목표가 아니라, 작명 추천 결과에 포함되는 한자의 뜻, 음, 유니코드, 획수, 발음오행, 자원오행이 일관되게 설명 가능한지를 기준으로 데이터를 구분했다.

---

## 2. 작업 목적

한자 작명 QA 시스템은 RAG 검색 결과를 기반으로 이름 후보를 추천한다. 따라서 검색 DB에 들어가는 한자 데이터는 다음 조건을 만족해야 한다.

- 한자와 한글 음이 명확해야 한다.
- 뜻 정보가 추천 문장에 사용할 수 있는 형태여야 한다.
- 유니코드가 한자 식별 기준으로 정리되어 있어야 한다.
- 획수 정보가 수리 판단에 사용할 수 있어야 한다.
- 발음오행과 자원오행이 일관되게 매핑되어 있어야 한다.
- ChromaDB와 LangGraph 파이프라인에서 사용할 수 있는 `document + metadata` 구조여야 한다.

기존 운영 기준 데이터는 `hanja_documents.json` 2,420건이었다. 여기에 포함되지 않은 한자 6,564건을 확장 후보군으로 정리하고, 전체 8,984건을 대상으로 운영 적합성을 비교할 수 있도록 구성했다.

---

## 3. 데이터 구성

작업 후 한자 데이터는 다음과 같이 구분했다.

| 구분 | 파일 | 건수 | 역할 |
|---|---|---:|---|
| A 후보군 | `data/processed/hanja_documents.json` | 2,420 | 정제 완료 운영 기본 후보군 |
| 확장 후보군 | `data/processed/hanja2_candidate_documents.json` | 6,564 | 원본 범위에서 추가 확보한 후보군 |
| B 후보군 | A 후보군 + 확장 후보군 | 8,984 | 전체 후보군 통합 검색 실험군 |
| C 후보군 | A 후보군 우선 검색 + 확장 후보군 보조 검색 | 2,420 + 6,564 | 메인과 후보군을 분리 운용하는 검토군 |

최종 운영 판단은 전체 후보군을 버리는 방식이 아니라, 정제 수준에 따라 운영 기본 데이터와 확장 검토 데이터를 분리하는 방식으로 정리했다.

---

## 4. 확장 후보군 JSON 정제

기존 2,420건에 포함되지 않은 한자를 `hanja2_candidate_documents.json`으로 정리했다.

이 파일은 기존 `hanja_documents.json`과 동일하게 `id`, `document`, `metadata` 구조를 사용한다. 이렇게 맞춘 이유는 확장 후보군도 필요할 경우 ChromaDB에 적재하거나 테스트용 컬렉션에서 검색할 수 있도록 하기 위해서다.

기준 구조는 다음과 같다.

```json
{
  "id": "hanja_OHE2-00001",
  "document": "한자 仮(가)은 뜻이 '거짓'인 인명용 한자이다. 유니코드는 U+4EEE, 획수는 6획, 발음오행은 목, 자원오행은 화이다.",
  "metadata": {
    "type": "hanja",
    "collection": "hanja_profiles",
    "source": "hanja2_candidate_documents.json",
    "profile_id": "OHE2-00001",
    "hangul": "가",
    "hanja": "仮",
    "unicode": "U+4EEE",
    "sound_meaning": "거짓",
    "strokes": 6,
    "sound_ohaeng": "목",
    "resource_ohaeng": "화",
    "is_person_name_hanja": true
  }
}
```

후보군 정제 과정에서는 `미확인`, `검증대기`처럼 사용자 답변에 그대로 노출되기 어려운 표현을 줄이고, 가능한 범위에서 실제 의미와 오행 정보를 반영했다.

---

## 5. ChromaDB 후보군 비교 설계

확장 후보군을 만들었다고 해서 바로 운영 DB에 합치는 것은 적절하지 않다고 판단했다. 검색 범위가 넓어지는 장점은 있지만, 추천 결과에 후보군이 직접 섞이면 설명 책임과 후처리 검증 부담이 커질 수 있기 때문이다.

따라서 기존 운영 ChromaDB는 건드리지 않고, 테스트용 ChromaDB를 별도로 구성하는 방향으로 설계했다.

| 테스트 방식 | 컬렉션 | 적재 데이터 | 목적 |
|---|---|---|---|
| Baseline | `hanja_base_test_col` | 2,420건 | 정제 완료군만 사용했을 때의 안정성 확인 |
| Expanded | `hanja_expanded_test_col` | 8,984건 | 전체 후보군 통합 시 검색 폭과 리스크 확인 |
| Hybrid | `hanja_base_test_col` + `hanja_candidate_test_col` | 2,420건 + 6,564건 | 메인 데이터와 후보군을 분리 검색하는 방식 확인 |

이 비교의 목적은 2,420건을 처음부터 고정하는 것이 아니라, 전체 후보군을 비교한 뒤 운영 기본값으로 어떤 구성이 가장 적합한지 확인하는 것이다.

---

## 6. 기존 파이프라인 유지

테스트는 새로운 작명 파이프라인을 만드는 방식이 아니라 기존 `naming_graph.py` 구조를 유지하는 방식으로 설계했다.

추가한 테스트 노트북은 다음과 같다.

```text
tests/한자_ChromaDB_운영후보군_비교검증.ipynb
```

노트북에서는 기존 LangGraph 흐름을 그대로 사용하되, 한자 검색에 사용하는 ChromaDB 컬렉션만 테스트용 컬렉션으로 바꾸어 A/B/C 결과를 비교할 수 있도록 했다.

이 방식은 운영 파이프라인을 수정하지 않고도 후보군별 검색 결과와 추천 품질 차이를 검증할 수 있다는 장점이 있다.

## 8. 최종 판단

전체 한자 후보가 8,984건이라고 해서 모두 운영 DB에 넣는 것이 최선은 아니었다.

작명 추천은 검색 가능한 한자 수보다 추천 결과의 근거 품질이 중요하다. A 후보군 2,420건은 뜻, 음, 유니코드, 획수, 발음오행, 자원오행이 정제되어 있어 운영 기본값으로 사용하기에 가장 안정적이었다.

B 후보군은 검색 범위를 넓힐 수 있지만, 후보군이 자동 추천 결과에 직접 섞이면 검증 정책이 추가로 필요하다. C 후보군은 보조 검색 방식으로 가능성이 있지만, 운영 정책과 노출 기준을 별도로 정해야 한다.

따라서 최종 판단은 다음과 같다.

- 운영 기본 DB는 A 후보군 2,420건을 사용한다.
- 확장 후보군 6,564건은 삭제하지 않고 별도 검토 계층으로 관리한다.
- B/C 후보군은 향후 운영 정책이 정해진 뒤 보조 검색 또는 확장 검토 용도로 활용한다.

## 10. 정리

이번 작업을 통해 한자 데이터 정제, 확장 후보군 분리, ChromaDB 비교 검증, 최종 운영 기준 문서화를 하나의 흐름으로 정리했다.

핵심은 전체 후보군을 줄이는 것이 아니라, 운영 가능한 데이터와 확장 검토 데이터를 명확히 분리하는 것이었다.

최종적으로 정제 완료 A 후보군 2,420건을 운영 기본값으로 두고, 나머지 6,564건은 향후 확장 검토 계층으로 유지하는 방향이 가장 안정적이라고 판단했다.
