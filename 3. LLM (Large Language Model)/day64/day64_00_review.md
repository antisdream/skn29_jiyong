# Day64 Review: 작명 AI 파이프라인 테스트 노트북 개선

## 1. 개요

이 문서는 작명 AI 프로젝트의 테스트 노트북을 개선한 내용을 정리한 기록이다. 주요 대상은 `jiyong.ipynb`이며, 원본 그래프 구현인 `src/graph/naming_graph.py`는 수정하지 않고 노트북 내부에서 모델, 프롬프트, 질의, 검증 출력을 실험할 수 있도록 구성했다.

이번 정리의 목적은 단순한 최종 답변 확인이 아니라, LLM 기반 작명 파이프라인이 어떤 근거와 도구 흐름을 통해 답변을 생성하는지 추적하는 것이다.

---

## 2. 작업 목적

작명 AI는 RAG 검색, 라우팅, 생성 모델, 후보군 검증이 함께 동작하는 구조다. 따라서 모델 출력만 확인하면 다음 문제를 파악하기 어렵다.

- 모델이 어떤 후보군을 참고했는지 확인하기 어렵다.
- RAG 후보와 최종 추천 이름의 연결 관계가 불명확하다.
- 순우리말, 한자어, 혼종어 추천 결과가 섞일 수 있다.
- 이름이 아닌 설명 단어가 후보군으로 잘못 추출될 수 있다.
- 모델별 응답 품질 차이를 같은 조건에서 비교하기 어렵다.
- 파이프라인이 실제로 어떤 노드와 Tool을 거쳤는지 확인하기 어렵다.

이를 해결하기 위해 테스트 노트북에 모델 비교, 파이프라인 추적, 후보군 검증, 렌더링 검증 기능을 보완했다.

## 4. 주요 개선 사항

### 4.1 모델 설정 분리

노트북에서 테스트할 GPT 모델을 직접 지정할 수 있도록 구성했다.

```python
AVAILABLE_TEST_MODELS = [
    "gpt-4.1-mini",
    "gpt-5.4-nano",
    "gpt-5.4-mini",
]
```

라우터 모델과 생성 모델은 각각 분리해서 테스트할 수 있다.

```python
ROUTER_MODEL = "gpt-5.4-mini"
GENERATE_MODEL = "gpt-5.4-mini"
```

이 구조를 통해 라우팅 성능과 답변 생성 품질을 분리해서 비교할 수 있다.

### 4.2 프롬프트 실험 셀 유지

원본 그래프의 프롬프트를 직접 수정하지 않고, 노트북 내부에서 프롬프트를 덮어써 실험할 수 있도록 했다.

| 설정 | 설명 |
|---|---|
| `USE_NOTEBOOK_PROMPTS = True` | 노트북 프롬프트 사용 |
| `USE_NOTEBOOK_PROMPTS = False` | 원본 그래프 프롬프트 사용 |

이 방식은 팀원별 프롬프트 실험 결과를 비교하고, 최종적으로 안정적인 프롬프트만 원본 코드에 반영하기 위한 구조다.

### 4.3 모델 동시 비교 셀 추가

같은 질의를 여러 모델에 동시에 실행하고 결과를 비교할 수 있도록 했다.

```python
COMPARE_GENERATE_MODELS = [
    "gpt-5.4-mini",
    "gpt-5.4-nano",
    "gpt-4.1-mini",
]
```

모델 목록에서 특정 줄을 주석 처리하면 비교 대상을 줄일 수 있다.

```python
COMPARE_GENERATE_MODELS = [
    "gpt-5.4-mini",
    # "gpt-5.4-nano",
    "gpt-4.1-mini",
]
```

비교 결과에는 모델명, 실행 시간, 사용 Tool, 반복 횟수, context 길이, 최종 답변이 함께 표시된다.

## 5. 파이프라인 시각화

작명 QA는 단일 LLM 호출이 아니라 그래프 기반 파이프라인으로 동작한다. 따라서 모델 비교 결과에 내부 실행 흐름을 표시하도록 했다.

예시 흐름은 다음과 같다.

```text
router -> internal_rag -> generate
```

단계별 로그에서는 다음 정보를 확인할 수 있다.

| 항목 | 설명 |
|---|---|
| step | 실행 순서 |
| node | 실행된 그래프 노드 |
| next action | 다음 액션 |
| iterations | 반복 횟수 |
| used tools | 사용된 Tool 목록 |
| collections | RAG 검색 컬렉션 |
| context length | 누적 context 길이 |

이 출력은 최종 사용자용 UI가 아니라 개발자가 파이프라인 동작을 검증하기 위한 도구다.

## 6. 후보군 검증 로직

### 6.1 순우리말 후보군 검증

순우리말 이름은 `urimalsam_col` 검색 결과를 바탕으로 추천된다. 따라서 최종 답변에 포함된 이름이 RAG 후보군에서 온 것인지 확인할 수 있도록 했다.

| 표시 상태 | 의미 |
|---|---|
| 후보 포함 | RAG 순우리말 후보에서 확인된 이름 |
| 후보 외 생성/파싱 확인 | 검색 후보와 불일치하거나 파싱 확인이 필요한 이름 |

예를 들어 RAG 후보에 `도운`, `슬찬`이 있고 최종 답변에 `윤도운`, `윤슬찬`이 포함되면 후보 포함으로 표시된다.

### 6.2 한자어 후보군 검증

한자어 이름은 완성 이름 후보를 그대로 가져오는 방식이 아니라, 인명용 한자 재료를 바탕으로 조합하는 방식에 가깝다.

따라서 한자어 이름에는 다음 기준을 적용했다.

| 표시 상태 | 의미 |
|---|---|
| 한자 재료 기반 조합 | 한자/한자음 후보를 바탕으로 생성된 이름 |

한자어 검증 화면에서는 한자 샘플 수, 전체 한자 수, 한자음 후보 수, 최종 조합 이름을 확인할 수 있다.

## 7. 후보군 파싱 보완

후보군 추출 과정에서 이름이 아닌 설명 단어가 후보로 섞이는 문제가 있었다.

예시:

```text
발음오행, 자원오행, 설명, 한자, 조합했습니다
```

위 단어들은 이름이 아니라 답변 설명에 포함된 메타 정보이므로 후보군에서 제외하도록 했다.

또한 여러 이름이 하나의 문자열로 붙어 나오는 경우도 처리했다.

```text
윤새윤도운윤슬찬
```

위와 같은 문자열은 다음처럼 분리한다.

```text
윤새, 윤도운, 윤슬찬
```

단, `윤도윤`처럼 이름 안에 성씨 음절이 자연스럽게 포함된 경우는 하나의 정상 이름으로 유지해야 한다. 이를 위해 정상적인 전체 이름 길이 안에서는 성씨 음절이 반복되어도 분리하지 않도록 조건을 추가했다.

## 8. 발생 오류와 해결

### 8.1 CSS 중괄호 오류

모델 비교 리포트 HTML을 출력하는 과정에서 다음 오류가 발생했다.

```text
NameError: name 'width' is not defined
```

원인은 Python f-string 내부에 CSS 중괄호가 그대로 들어간 것이었다.

문제가 된 형태:

```python
.wgs-funnel-table { width: 100%; }
```

f-string 내부에서 `{}`는 Python 표현식으로 처리되기 때문에 CSS의 `width`를 변수로 해석하려고 했다.

수정한 형태:

```python
.wgs-funnel-table {{ width: 100%; }}
```

CSS 중괄호를 이스케이프하여 HTML 리포트가 정상 렌더링되도록 수정했다.

### 8.2 리스트 렌더링 오류 가능성

파이프라인 경로는 내부적으로 리스트 형태로 관리된다.

```python
["router", "internal_rag", "generate"]
```

이 값을 HTML에 그대로 넣으면 문자열 변환 과정에서 오류가 발생할 수 있다. 따라서 표시용 문자열로 변환했다.

```text
router -> internal_rag -> generate
```

## 11. 모델별 테스트 관찰 내용

모델 비교 셀에서는 동일 질의를 여러 생성 모델에 입력하여 출력 형식과 추천 결과를 비교했다. 아래 내용은 제한된 테스트 과정에서 관찰된 결과이며, 특정 모델의 우열을 확정하기 위한 평가는 아니다.

| 모델 | 관찰 내용 | 보완 필요 항목 |
|---|---|---|
| `gpt-4.1-mini` | 여러 개 추천 요청에서 요청 개수를 비교적 잘 맞춘 출력이 있었다. | 한자어 이름 조합의 자연스러움과 실제 이름감 검토 필요 |
| `gpt-5.4-mini` | 일부 테스트에서 요청 개수보다 적은 결과가 출력되거나 외자 형태의 결과가 나왔다. | 이름 개수, 이름 길이, 유형별 분리 조건 검증 필요 |
| `gpt-5.4-nano` | 비교 대상으로 설정했지만 충분한 출력 사례가 누적되지 않았다. | 동일 조건의 반복 테스트 필요 |

이번 테스트에서는 모델별로 출력 형식과 이름 추천 방식에 차이가 있음을 확인했다. 다만 테스트 횟수가 제한적이므로 특정 모델을 기준 모델로 확정하기보다는, 동일 질의에 대한 반복 테스트와 후보군 검증 로직을 함께 사용해 평가하는 방식이 필요하다.
