## 3. Notion 업로드용 통합 문서 정리

프로젝트가 많이 진행되면서 `docs` 폴더 안에 Markdown 문서와 PDF 문서가 많이 쌓였다.  
이 문서들을 Notion에 하나씩 올리면 흐름이 끊기기 때문에, 전체 프로젝트 상황을 한눈에 볼 수 있도록 하나의 Markdown 파일로 묶었다.

생성한 파일은 다음 위치에 저장했다.

```text
C:\python-src\3rd_PROJECT\third_project\SKN29_3rd_Naming_QA_Notion_Integrated.md
```

포함한 문서 범위는 다음과 같다.

| 문서 종류 | 개수 | 처리 방식 |
|---|---:|---|
| Markdown 문서 | 22개 | 원문을 부록 형태로 통합 |
| PDF 문서 | 2개 | 페이지 수, 원본 경로, 추출 가능한 요약 포함 |
| 프로젝트 요약 | 1개 | 직접 정리 |
| 파이프라인 요약 | 1개 | 데이터 수집부터 LLM 답변 생성까지 흐름 정리 |

통합 문서 안에는 다음 섹션을 넣었다.

- 프로젝트 한눈에 보기
- 전체 파이프라인
- 문서 카탈로그
- 팀 작업 흐름 기준 정리
- Notion 업로드 후 추천 페이지 구조
- Markdown 원문 부록
- PDF 참고자료 요약

이 작업을 하면서 느낀 점은, 프로젝트가 커질수록 기능 구현만큼이나 **문서의 흐름을 정리하는 작업**이 중요하다는 점이었다.  
특히 우리 프로젝트는 ChromaDB, Neo4j, SQLite, LangGraph, MCP, LLM이 모두 들어가기 때문에, 문서가 흩어져 있으면 팀원 간 이해 수준이 금방 달라질 수 있다.

## 4. Qwen3.5-4B 성능 문제 분석

팀장이 Docker 로컬 서버에서 `Qwen3.5-4B` 모델을 띄우고 Web AI 형태로 QA 테스트를 진행했다.  
하지만 실제 테스트 결과, 답변 생성 시간이 길고 성능도 만족스럽지 않았다.

처음에는 파인튜닝으로 해결할 수 있을지 고민했지만, 분석해보니 바로 파인튜닝에 들어가기 전에 먼저 확인해야 할 부분이 있었다.

### 4-1. 느린 원인

현재 구조에서는 LLM이 한 번만 호출되는 것이 아니라, 다음과 같은 흐름으로 여러 번 호출될 수 있다.

```text
사용자 질문
→ LangGraph Router 판단
→ MCP Tool 실행
→ 다시 Router 판단
→ 최종 답변 생성
```

즉, 느린 원인은 단순히 모델 하나가 느린 것이 아니라 다음 요인이 겹친 결과였다.

| 원인 | 설명 |
|---|---|
| Router LLM 호출 | Tool 선택에도 LLM을 사용 |
| ReAct 반복 | Tool 실행 후 다시 LLM 판단으로 돌아감 |
| 긴 context | `num_ctx=32768`처럼 큰 컨텍스트 설정 |
| 긴 출력 | 답변 길이에 대한 제한이 약함 |
| thinking 모드 | Qwen 계열 모델의 내부 추론 시간이 길어질 수 있음 |

### 4-2. 속도 개선 방향

속도 개선을 위해서는 먼저 파이프라인 구조를 가볍게 만들어야 한다.

| 구간 | 개선 방향 |
|---|---|
| Router | 규칙 기반 분기 우선 적용 |
| ChromaDB | 필요한 컬렉션만 검색 |
| 검색 결과 | `n_results`를 상황별로 조정 |
| context | 최종 답변에 필요한 정보만 압축 |
| LLM 호출 | 단순 조회는 LLM 없이 결과 반환 |
| 출력 | `num_predict` 또는 `max_output_tokens` 제한 |

추천 설정은 다음과 같이 정리했다.

```python
ChatOllama(
    model="qwen3.5:4b",
    temperature=0.4,
    top_p=0.85,
    top_k=20,
    num_ctx=8192,
    num_predict=1000,
    repeat_penalty=1.05,
)
```

## 5. Qwen3.5-4B, GPT-5.4-nano, GPT-5.5 비교

Qwen3.5-4B만 계속 최적화할지, 아니면 OpenAI 모델을 일부 사용할지도 비교했다.

중요한 차이는 다음과 같다.

| 모델 | 장점 | 단점 | 적합한 역할 |
|---|---|---|---|
| Qwen3.5-4B | 로컬 실행 가능, 비용 절감, QLoRA 파인튜닝 가능 | 속도와 품질이 장비와 튜닝에 크게 의존 | 오프라인 백업, 파인튜닝 실험 |
| GPT-5.4-nano | 빠르고 저렴한 API 모델, 라우팅/요약에 적합 | 복잡한 상담형 답변은 한계 가능 | 빠른 Router, 기본 QA |
| GPT-5.5 | 복합 추론과 자연어 품질이 가장 좋음 | 비용이 높고 effort 설정에 따라 느려질 수 있음 | 최종 작명 상담, 고품질 답변 |

내 판단으로 가장 현실적인 구조는 하나의 모델만 쓰는 것이 아니라 역할을 나누는 방식이었다.

```text
규칙 기반 Router
→ ChromaDB / Neo4j / SQLite / API Tool 실행
→ 기본 답변은 GPT-5.4-nano
→ 복잡한 작명 상담은 GPT-5.5
→ 로컬 백업과 파인튜닝 실험은 Qwen3.5-4B
```

이렇게 하면 비용, 속도, 품질을 모두 어느 정도 균형 있게 가져갈 수 있다.

## 6. 이름 추천 품질 문제 트러블슈팅

오늘 가장 중요한 품질 이슈 중 하나는 실제 Web AI 테스트에서 나온 이름 추천 결과였다.

테스트 질문은 다음과 같았다.

```text
최씨 성을 가진 아이 이름을 남자, 여자 각각 5명씩 추천해줘.
```

그런데 결과로 다음과 같이 발음하기 어렵거나 이름답지 않은 후보가 나왔다.

```text
최포유
최점거
최포우
```

또한 `최지은`이라는 이름이 반복적으로 등장하는 문제도 있었다.

### 6-1. 원인 1 - LLM이 이름을 직접 만들어버림

현재 파이프라인은 이름 후보를 시스템이 먼저 만들고 검증한 뒤 LLM이 설명하는 구조가 아니라,  
LLM이 검색된 한자 독음과 일반 지식을 보고 직접 이름을 조합할 가능성이 있다.

특히 질문에 `한자`, `오행`, `획수`, `순우리말` 같은 키워드가 명확히 들어가지 않으면 적절한 컬렉션 검색이 아니라 fallback 검색으로 흐를 수 있다.

그 결과 한자 독음으로는 존재하지만 실제 이름으로는 어색한 조합이 추천될 수 있다.

### 6-2. 원인 2 - `johab.json`은 이름 선호 음절표가 아님

`data/raw/reference/johab.json`은 상용한자에 있는 한글 음절 목록이다.  
즉, 이 파일은 “이름으로 자연스러운 음절” 목록이 아니라 “한자 독음으로 존재하는 음절” 목록이다.

따라서 이 데이터만 통과 기준으로 삼으면 `포`, `점`, `거` 같은 음절도 이름 후보로 살아남을 수 있다.

### 6-3. 원인 3 - `최지은` 반복

`최지은`이 반복되는 이유도 어느 정도 확인할 수 있었다.

`paper_documents.json` 안에는 최근 한국 이름 경향을 다루는 논문 청크가 들어 있고, 그 안에 `지은`, `지혜`, `지현`, `지영`, `지원` 같은 이름이 등장한다.  
또한 여자 이름에서 첫 글자 `지`, 끝 글자 `은`이 선호된다는 내용도 포함되어 있다.

따라서 모델은 이 내용을 보고 `지은`을 안정적인 여자 이름 후보로 과하게 일반화했을 가능성이 있다.

### 6-4. 해결 방향

이 문제는 매개변수만으로는 완전히 해결하기 어렵다.  
이름 추천은 반드시 다음 구조로 바뀌어야 한다.

```text
후보 생성
→ 후보 검증
→ 후보 랭킹
→ LLM 설명 생성
```

LLM에게 이름을 직접 만들게 하는 것이 아니라, 시스템이 검증한 후보 목록 안에서만 설명하게 해야 한다.

필요한 검증은 다음과 같다.

| 검증 항목 | 설명 |
|---|---|
| 발음 자연스러움 | `점거`, `포우` 같은 어색한 조합 제거 |
| 실제 이름 빈도 | 출생신고 상위 이름 통계와 대조 |
| 성별 선호도 | 남자/여자 선호 음절 반영 |
| 부정 어감 제거 | 일상 단어, 부정적 의미 단어 제외 |
| 인명용 한자 검증 | Neo4j 또는 한자 데이터 기준 검증 |
| 수리/오행 검증 | SQLite/DB server와 오행 데이터 기준 검증 |

프롬프트도 다음처럼 바꿔야 한다.

```text
아래 [검증된 후보 목록]에 없는 이름은 절대 생성하지 마세요.
발음이 어색한 이름, 일상 단어처럼 들리는 이름, 부정적 의미가 있는 이름은 추천하지 마세요.
각 추천 이름은 후보 목록의 근거를 사용해서만 설명하세요.
```

## 7. 오늘 느낀 점

오늘 가장 크게 느낀 점은 **RAG 프로젝트는 모델만 좋다고 해결되지 않는다**는 것이다.

처음에는 Qwen3.5-4B의 성능이 낮으니 파인튜닝이나 더 큰 모델 사용만 생각했지만, 실제 문제를 뜯어보니 다음 요소가 모두 연결되어 있었다.

- 어떤 데이터를 검색할 것인가
- 검색 결과를 얼마나 넣을 것인가
- LLM이 직접 생성해도 되는 영역과 절대 생성하면 안 되는 영역을 어떻게 나눌 것인가
- Tool 결과를 어떻게 검증할 것인가
- 결과 품질을 어떻게 검증하고 개선할 것인가

특히 이름 추천 품질 문제를 보면서, 작명 시스템은 단순 생성형 AI가 아니라 **후보 생성, 후보 검증, 후보 랭킹, 근거 설명**이 분리되어야 한다는 점을 확인했다.

LLM은 이름을 마음대로 만드는 역할이 아니라, 검증된 후보를 사용자에게 이해하기 쉽게 설명하는 역할을 맡아야 한다.
