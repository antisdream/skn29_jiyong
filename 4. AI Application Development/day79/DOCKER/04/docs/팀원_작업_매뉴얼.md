# 4차 프로젝트 — 팀원용 작업 매뉴얼

> 이 문서는 **로컬 PC에서** 진행하는 작업만 다룹니다. EC2 서버 설정은 팀장이 별도로 관리하며, 팀원은 EC2에 직접 접속할 필요가 없습니다.
>
> **이 문서의 모든 명령어는 본인 PC의 터미널(Git Bash / PowerShell / 터미널 앱 등)에서, `SKN29-4th-4Team` 폴더 안에서 실행합니다.** (1번에서 클론한 뒤 그 폴더로 들어간 상태가 계속 유지된다고 가정) EC2나 다른 서버에 접속해서 입력하는 명령어는 이 문서에 없습니다.

```
당신의 PC (이 매뉴얼)              GitHub                    EC2 (팀장 관리)
──────────────────────           ──────────                ──────────────
코드 작성 → 로컬 테스트   →   feature 브랜치 push   →   (팀장이 main 병합)
                                                              ↓
                                                    자동 배포 (GitHub Actions)
```

---

## 0. 사전 준비물

- **Docker Desktop** 설치 및 실행 중인 상태
- **Git** 설치
- 저장소 접근 권한 (GitHub `SKN29-4th-4Team` 콜라보레이터로 초대받아야 함)

---

## 1. 저장소 클론

```bash
cd <원하는 작업 폴더>
git clone https://github.com/Somber-7/SKN29-4th-4Team.git
cd SKN29-4th-4Team
```

---

## 2. 로컬 환경 파일 준비

`.env.local.example`을 복사해서 `.env.local`을 만듭니다 (실제 값 입력, 이 파일은 git에 올라가지 않음):

```bash
cp .env.local.example .env.local
```

`OPENAI_API_KEY` 등은 팀 채널에서 공유받은 값을 넣으세요. 나머지(`DB_*`, `NEO4J_*`, `DJANGO_SECRET_KEY`)는 예시 파일 그대로 두거나 아무 문자열이나 넣어도 됩니다 (로컬 전용 값이라 보안/공유 무관, 다 같아도 상관없음).

---

## 3. 로컬에서 전체 스택 실행

```bash
docker compose --env-file .env.local -f docker-compose.local.yml up --build
```

처음 실행은 이미지 빌드 때문에 몇 분 걸립니다. 아래처럼 여러 컨테이너가 뜨는 게 보이면 정상입니다:

```
postgres-local  ... Up
neo4j           ... Up
django          ... Up
fastapi         ... Up
nginx           ... Up
```

**최초 1회만** 실행 — 위 `up --build` 명령이 계속 실행 중인 터미널은 그대로 두고, **새 터미널 창을 열어서 `cd`로 같은 `SKN29-4th-4Team` 폴더에 다시 들어간 뒤** 아래 명령을 입력:

```bash
docker compose --env-file .env.local -f docker-compose.local.yml exec django python manage.py migrate
```

## 4. 확인

브라우저에서 `http://localhost/` 접속 → `{"status": "ok", "service": "django"}` 응답이 보이면 성공.

```bash
curl http://localhost/api/health   # FastAPI 응답 확인
```

### Neo4j 그래프 데이터 채우기

`docker-compose.local.yml`의 `neo4j` 컨테이너는 **매번 새로 만들 때 완전히 빈 상태**로 시작합니다. RDS/ChromaDB와 달리 Neo4j는 미리 채워진 데이터가 볼륨에 같이 들어있지 않아서, 한자-오행 관계 조회(`graph_db` 기능)를 테스트하려면 아래 스크립트로 **직접 한 번 채워줘야** 합니다.

**필요한 경우만 실행하세요** — 프론트엔드 화면 작업이나 RAG(ChromaDB) 검색만 테스트한다면 이 단계는 건너뛰어도 됩니다. 채워두지 않으면 `graph_db` 관련 질의만 빈 결과가 나옵니다.

```bash
# 1. 연결 확인 (데이터를 쓰기 전에 먼저 확인, 아직 아무것도 안 씀)
docker compose --env-file .env.local -f docker-compose.local.yml exec fastapi python src/graph/index_hanja_neo4j.py --check-connection

# 2. 실제로 데이터 적재 (아래처럼 진행률이 표시됨, 완료까지 1분 이내)
docker compose --env-file .env.local -f docker-compose.local.yml exec fastapi python src/graph/index_hanja_neo4j.py --execute
```

`--execute` 실행 시 이런 로그가 나오며 끝까지 진행되면 정상입니다:

```
Indexed 500/2420 Hanja nodes into Neo4j.
Indexed 1000/2420 Hanja nodes into Neo4j.
...
Indexed 2420/2420 Hanja nodes into Neo4j.
Neo4j Hanja indexing completed.
```

**주의:** 7번(`docker compose down -v`)으로 볼륨까지 삭제하면 Neo4j 데이터도 같이 지워집니다. 다시 `up`한 뒤에는 이 단계를 한 번 더 실행해야 합니다.

## 5. 작업 방식

```bash
git checkout -b feature/내작업이름
# 코드 수정
# 위 3번 명령으로 로컬에서 다시 테스트 (변경사항 반영하려면 --build 다시)
git add .
git commit -m "작업 내용"
git push origin feature/내작업이름
```

이후 GitHub에서 **Pull Request** 생성 → 팀장이 리뷰 후 `main`에 병합 → 자동으로 EC2에 반영됩니다.

---

## 6. 코드 작업 위치

**`naming` 앱은 이미 만들어져서 공유 저장소에 있습니다. 절대 `python manage.py startapp naming`을 다시 실행하지 마세요** — 각자 새로 만들면 구조가 갈라져서 병합할 때 전부 충돌납니다. 이미 있는 `webapp/naming/` 위에 기능을 추가하는 방식으로 작업하세요.

| 담당 | 작업 위치 |
|------|----------|
| Django 모델 | `webapp/naming/models.py` |
| Django 뷰/로직 | `webapp/naming/views.py` |
| URL 라우팅 | `webapp/naming/urls.py` (각자 `path()` 추가) |
| FastAPI 응답 형식/기능 추가 | `fastapi_app/main.py` |
| 화면 스타일/레이아웃 | `webapp/naming/static/naming/`, `webapp/naming/templates/naming/` (새 폴더 생성) |
| Nginx 라우팅 | `deploy/nginx/nginx.conf` |

모델을 추가/수정했다면 반드시 마이그레이션 파일도 같이 커밋하세요:

```bash
docker compose --env-file .env.local -f docker-compose.local.yml exec django python manage.py makemigrations naming
docker compose --env-file .env.local -f docker-compose.local.yml exec django python manage.py migrate
git add webapp/naming/migrations/
```

`webapp/config/urls.py`의 `path('', health)`가 현재 루트(`/`)를 차지하고 있습니다. 실제 작명 홈페이지 뷰가 준비되면 이걸 대체하거나 `include('naming.urls')`로 바꾸면 됩니다 (파일에 주석으로 안내되어 있음). 이 부분을 바꿀 땐 다른 팀원과 미리 조율하세요 — 같은 파일을 동시에 건드리면 충돌 확률이 높습니다.

---

## 7. 종료

```bash
docker compose --env-file .env.local -f docker-compose.local.yml down
```

컨테이너만 내려가고, DB 데이터(볼륨)는 남아있어서 다음에 다시 `up` 하면 이어서 쓸 수 있습니다. 완전히 초기화하고 싶으면 `down -v` (볼륨까지 삭제).

## 8. 트러블슈팅

| 증상 | 원인 | 해결 |
|------|------|------|
| `docker compose` 명령 안 먹힘 | Docker Desktop 꺼짐 | Docker Desktop 실행 확인 |
| Django가 DB 연결 실패 | `.env.local` 안 만듦 또는 `migrate` 안 함 | 2번, 3번 단계 다시 확인 |
| 포트 충돌 (80, 5432 등) | 다른 프로그램이 이미 사용 중 | 해당 프로그램 종료하거나 `docker-compose.local.yml`의 포트 번호 수정 |
| 코드 수정했는데 반영 안 됨 | 이미지 재빌드 안 함 | `docker compose --env-file .env.local -f docker-compose.local.yml up --build` 다시 |
| `NEO4J_USER`/`NEO4J_PASSWORD` not set 경고, `neo4j:7687` DNS resolve 실패 | `--env-file .env.local`을 빼고 실행함 (Docker Compose는 `${VAR}` 치환을 `.env`라는 이름의 파일만 자동으로 읽어서, `.env.local`은 매번 `--env-file`로 직접 지정해야 함) | 이 문서의 모든 명령어에 `--env-file .env.local`이 포함되어 있는지 확인 |
| Neo4j `AuthError: Unauthorized` (연결은 되는데 인증 실패) | 위 문제를 먼저 겪었던 경우 — Neo4j는 최초 실행 시에만 `NEO4J_AUTH`를 반영하고, 이후엔 볼륨에 저장된 예전 인증 정보를 그대로 씀. `--env-file` 없이 한 번이라도 실행했다면 빈 인증으로 이미 초기화된 상태 | `docker compose --env-file .env.local -f docker-compose.local.yml down -v`로 볼륨까지 삭제 후 재시작, migrate와 Neo4j 인덱싱 다시 실행 |
