### AI 에이전트 실행 지침 (이 문서만 보고 작업하는 경우)

이 문서를 참고해 작업하는 AI 에이전트는 실행 전 아래를 지킨다.

1. **경로를 먼저 실측한다.** 이 문서는 백엔드 레포를 `Desktop\project4\SKN29-4th-4Team`으로 가정한다. 작업 시작 전 `ls`로 실제 폴더명·구조를 확인하고, 다르면 실측값을 우선한다. 두 폴더(demo3, project4) 모두 접근 불가능하면 작업을 중단하고 사용자에게 폴더 접근 권한을 요청한다.
2. **문서의 백엔드 서술은 추정이다.** demo3 쪽 내용(client.ts, types.ts, vite.config.ts)은 실제 코드 기준으로 검증됐지만, 백엔드 쪽(nginx.conf 내용, compose 서비스명 django/fastapi/neo4j, deploy/ 구조)은 분석 문서 기반 추정이다. STEP 2 이후는 반드시 실제 파일을 읽고 대조한 뒤 수정한다. 서비스명이 다르면(예: `web` vs `django`) nginx.conf의 `proxy_pass` 호스트명도 실제 compose 서비스명으로 맞춘다.
3. **파괴적 작업은 확인 후 진행한다.** demo3 원본 삭제, `git push`, 기존 파일 덮어쓰기(nginx.conf·Dockerfile 교체)는 실행 전 사용자에게 확인받는다. 병합은 복사(robocopy) 방식이므로 demo3 원본은 자동으로 보존된다.
4. **1차 범위만 요청받았으면 STEP 1에서 멈춘다.** STEP 1 검증(빌드 성공 + dev 서버 화면 확인)을 통과하면 완료 보고하고, STEP 2 이후를 임의로 진행하지 않는다.
5. **오류가 나면 9장을 먼저 찾는다.** 증상이 9장에 있으면 해당 해결책을 적용하고, 없으면 원인 파악 결과와 함께 사용자에게 보고한다.

백엔드 API가 아직 구현되지 않았으므로 작업을 두 단계로 나눈다.

## 1. 최종 결정 요약

두 분석 문서의 결론을 종합한 확정 사항이다. v1의 방안 B(Django가 SPA 서빙)는 v2에서 폐기됐고, 아래가 최종안이다.

| 항목 | 최종 결정 | 근거 |
|---|---|---|
| 기준 레포 | **project4의 `SKN29-4th-4Team`이 기준.** demo3는 그 안의 `frontend/`로 편입 | 배포 설정(compose·deploy·docs)이 이미 백엔드 레포에 있음. 역방향은 배포 재작성 손해 |
| 서빙 구조 | **방안 A: nginx가 React 빌드(dist/)를 직접 서빙** | 구조 깔끔, 프론트 계약 그대로 사용, CORS 없음 |
| MVT 평가 대응 | Django Admin(User·History·FAQ) + API도 FBV/CBV + Django Form 검증으로 작성 | T(템플릿)는 Admin이, M·V·폼 검증은 사용자 기능이 실증 |
| 라우팅 | `/api/*` → Django(8000), `/naming-api/*` → FastAPI(8001), `/` → dist 정적 | 프론트 client.ts의 계약과 일치 |
| 인증 | Django 세션 + CSRF 쿠키 방식 | client.ts가 이미 `credentials: "include"` 구현 |
| 범위 | 인증·작명 생성·**이력(History)** 필수 / 인사이트·FAQ API·React 관리자 화면 제거 | History는 ORM·CRUD 채점 근거라 필수로 승격(v2 §3) |
| FastAPI 계약 | `/names/generate`를 신설하고 LangGraph에 Structured Output 강제 | 기존 `/ask`(자유 텍스트)로는 결과 카드 렌더링 불가 |
| 깃 이력 | 파일 복사 후 단일 커밋 (subtree 사용 안 함) | 이력 보존의 실익 없음, 리뷰 깔끔 |

### 최종 아키텍처

```
브라우저 (React SPA)
   │
nginx :80 ─── /              → dist/ 정적 서빙 (React 빌드, SPA fallback)
   ├────────── /api/*        → django:8000  (gunicorn/WSGI · 세션인증 · Postgres)
   │                            └ /admin/   → Django 기본 Admin
   ├────────── /naming-api/* → fastapi:8001 (LangGraph · Neo4j, proxy_read_timeout 120s)
   └────────── /static/*     → Django collectstatic (admin 정적 파일)
```

## 3. STEP 1 — 폴더 병합 (demo3 → frontend/) 【1차 · 지금 진행】

### 3.1 이동 규칙

demo3의 파일을 아래 규칙대로 옮긴다. **`node_modules/`와 `dist/`는 절대 옮기지 않는다** — 옮기면 복사에만 수십 분 걸리고 깃 레포가 오염된다.

| demo3 항목 | 처리 |
|---|---|
| `src/`, `public/`, `index.html`, `vite.config.ts`, `tsconfig.json`, `postcss.config.mjs`, `package.json`, `package-lock.json`, `.env.example` | `frontend/`로 이동 |
| `node_modules/`, `dist/` | **이동 금지** (병합 후 `frontend/`에서 `npm ci`로 재생성) |
| `_deprecated/`, `.claude/` | 이동하지 않음 |
| `화면설계서_최종.md`, 연결 전략 문서들, 이 가이드 | 레포의 `docs/`로 이동 |
| `README.md` | `frontend/README.md`로 이동 |

### 3.2 병합 명령 (PowerShell)

바탕화면에서 PowerShell을 열고 아래를 순서대로 실행한다. `robocopy`는 `node_modules`·`dist`를 제외하고 복사한다.

```powershell
$repo = "$env:USERPROFILE\Desktop\project4\SKN29-4th-4Team"
$demo = "$env:USERPROFILE\Desktop\demo3"

# 1) frontend/, docs/ 생성
New-Item -ItemType Directory -Force "$repo\frontend", "$repo\docs" | Out-Null

# 2) 소스 복사 (node_modules, dist, _deprecated, .claude 제외)
robocopy $demo "$repo\frontend" /E /XD node_modules dist _deprecated .claude /XF "화면설계서_최종.md" "프론트-백엔드_연결방안_분석.md" "프론트-백엔드_연결전략_v2.md" "프론트-백엔드_통합_최종가이드.md"

# 3) 문서는 docs/로
Copy-Item "$demo\화면설계서_최종.md","$demo\프론트-백엔드_*.md" "$repo\docs\"
```

> robocopy 종료 코드 1~7은 정상(복사 성공)이다. 8 이상만 오류다.

### 3.3 .gitignore / .dockerignore 추가

레포 루트 `.gitignore`에 아래 줄을 추가한다. 이 단계를 빼먹으면 `git add .` 한 번에 수만 개 파일이 스테이징된다.

```gitignore

```

레포 루트에 `.dockerignore`를 새로 만든다. 없으면 도커 빌드 컨텍스트가 수백 MB로 커져 빌드가 느려진다.

```

data/
*.md
```

## 4. STEP 2 — 도커·nginx 통합 【2차 · 백엔드 구현 후】

### 4.1 nginx Dockerfile을 multi-stage로 교체

nginx 이미지가 React 빌드를 포함하도록 `deploy/nginx/Dockerfile`을 아래로 교체한다. `package*.json`만 먼저 COPY하는 이유는 도커 레이어 캐시 활용이다 — 의존성이 안 바뀌면 `npm ci`를 재실행하지 않아 재빌드가 수 분에서 수 초로 줄어든다.

```dockerfile
# 1단계: React 빌드
FROM node:20-alpine AS build
WORKDIR /app
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ ./
RUN npm run build

# 2단계: nginx + 정적 파일
FROM nginx:alpine
COPY deploy/nginx/nginx.conf /etc/nginx/conf.d/default.conf
COPY --from=build /app/dist /usr/share/nginx/html
```

### 4.2 compose의 빌드 컨텍스트를 레포 루트로 변경

`docker-compose.4th.yml`과 `docker-compose.local.yml`의 nginx 서비스를 수정한다. **context를 루트로 바꾸지 않으면 `frontend/`가 빌드 컨텍스트 밖이라 COPY가 실패한다** — 병합 직후 가장 먼저 밟는 오류다(9장 B-1).

```yaml
nginx:
  build:
    context: .                        # 기존 ./deploy/nginx 에서 변경
    dockerfile: deploy/nginx/Dockerfile
```

같은 파일의 neo4j 서비스에 메모리 캡도 추가한다 (없으면 저사양 EC2에서 OOM, 9장 F-2).

```yaml
neo4j:
  environment:
    - NEO4J_server_memory_heap_max__size=1G
    - NEO4J_server_memory_pagecache_size=512M
```

### 4.3 nginx.conf 라우팅 교체

현재 nginx.conf는 `/api/`를 FastAPI로 보낸다. **프론트 계약은 `/api → Django`이므로 그대로 두면 로그인 요청이 FastAPI로 가서 404가 난다.** 아래로 교체한다.

```nginx
server {
    listen 80;

    # React SPA (빌드 산출물)
    location / {
        root /usr/share/nginx/html;
        try_files $uri /index.html;      # SPA 라우팅 fallback — 없으면 새로고침 시 404
    }

    # Django (인증·마이페이지·이력)
    location /api/ {
        proxy_pass http://django:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # Django Admin
    location /admin/ {
        proxy_pass http://django:8000;
        proxy_set_header Host $host;
    }

    # FastAPI (작명 생성 · LLM)
    location /naming-api/ {
        proxy_pass http://fastapi:8001/;   # 끝 슬래시: /naming-api/ 접두사 제거 후 전달
        proxy_set_header Host $host;
        proxy_read_timeout 120s;           # LLM 응답 대기 — 없으면 60초에 504
        limit_req zone=naming burst=5;     # LLM 비용 보호
    }

    # Django collectstatic (admin 정적 파일)
    location /static/ {
        alias /staticfiles/;               # compose에서 django 볼륨과 공유
    }
}
```

`http` 블록(또는 conf 최상단)에 rate limit zone을 선언한다.

```nginx
limit_req_zone $binary_remote_addr zone=naming:10m rate=5r/m;
```

> `proxy_pass` 끝 슬래시 유무에 따라 접두사 제거 여부가 달라진다. FastAPI 라우터가 `/names/generate`로 정의돼 있으면 위처럼 끝 슬래시를 붙여 `/naming-api/`를 떼고 전달한다. FastAPI 쪽에 `root_path="/naming-api"` 프리픽스를 쓰기로 했다면 슬래시를 빼고 그대로 전달한다. **팀에서 한 가지로 통일하고 시작한다.**

### 4.4 기동 검증

```powershell
cd "$repo"
docker compose -f docker-compose.local.yml up --build
```

브라우저에서 아래 3가지가 모두 되면 STEP 2 완료다.

1. `http://localhost` → React 랜딩 화면이 뜬다 (mock 모드 동작)

이 시점의 스크린샷과 `docker compose ps` 로그를 저장한다 — 테스트 보고서의 "컨테이너 재현성" 검증 자료가 된다.

## 5. STEP 3 — FastAPI 작명 API 연결 【2차】

### 5.1 계약(스키마) 확정 — 최우선 작업

프론트는 `POST /naming-api/names/generate`에 `NameRequest`를 보내고 `NameResult[]`를 기대한다. 기존 `/ask`(자유 텍스트 QA)와는 의미가 다르므로, **LangGraph 최종 노드에 Structured Output을 강제한 새 엔드포인트를 만든다.** 자유 텍스트를 정규식으로 파싱하는 방식은 금지 — 형식이 조금만 흔들려도 결과 화면 전체가 깨진다.

프론트 `frontend/src/app/types.ts` 기준 계약 (이 파일이 유일한 진실 원천이다):

```typescript
// 요청
type NameRequest =
  | { type: "natural"; query: string }
  | { type: "structured"; lastName: string; gender?: "남자" | "여자";
      elements?: string[]; strokeRange?: string; meaning?: string };

// 응답 (배열로 반환)
interface NameResult {
  id: number;
  lastName: CharBreakdown;          // { char, reading, ... }
  hanja: string;
  hangul: string;
  ruby: CharBreakdown[];
  sukgyeok: string;
  sukgyeokDetail: SukgyeokDetail[];
  sources: { type: "hanja" | "suri" | "beopryeong" | "nonmun"; label: string }[];
}
```

### 5.2 FastAPI 구현

`fastapi_app`에 Pydantic 모델과 엔드포인트를 추가한다. 필드명은 types.ts와 **camelCase 그대로** 맞춘다 (snake_case로 만들면 프론트 파싱이 전부 어긋난다 — 9장 D-4).

```python
from pydantic import BaseModel, Field
from typing import Literal, Optional

class CharBreakdown(BaseModel):
    char: str
    reading: str

class NameResult(BaseModel):
    id: int
    lastName: CharBreakdown
    hanja: str
    hangul: str
    ruby: list[CharBreakdown]
    sukgyeok: str
    sukgyeokDetail: list[dict]
    sources: list[dict]

@app.post("/names/generate", response_model=list[NameResult])
async def generate_names(req: dict):          # NameRequest는 union이라 dict로 받고 내부 분기
    try:
        results = await run_naming_graph(req)  # LangGraph 호출, structured output 강제
        return results
    except TimeoutError:
        raise HTTPException(504, detail={"message": "작명 생성 시간이 초과되었습니다."})
    except Exception:
        raise HTTPException(502, detail={"message": "작명 생성에 실패했습니다."})

@app.get("/names/sample-preview", response_model=list[NameResult])
async def sample_preview():
    ...
```

LangGraph 최종 노드에서는 `llm.with_structured_output(NameResultList)`로 스키마를 강제한다. 에러 응답은 반드시 `{"message": ..., "detail": ...}` 형태를 유지한다 — 프론트 `ApiError`가 이 형식을 기대한다.

### 5.3 검증

컨테이너 기동 후 아래 요청이 NameResult 배열 JSON을 반환하면 완료다.

```bash
curl -X POST http://localhost/naming-api/names/generate \
  -H "Content-Type: application/json" \
  -d '{"type":"structured","lastName":"김","gender":"남자"}'
```

응답의 각 원소가 `id, lastName, hanja, hangul, ruby, sukgyeok, sukgyeokDetail, sources` 키를 모두 가지면 계약 일치다. 하나라도 빠지면 프론트 결과 카드가 빈 값으로 렌더링된다.

## 6. STEP 4 — Django 인증 + CSRF 【2차】

### 6.1 Django 쪽 작업

DRF 없이 Django 기본 세션 + FBV로 구현한다. 프론트가 기대하는 엔드포인트는 `POST /api/auth/login`, `POST /api/auth/signup`, `POST /api/auth/forgot-password`다.

`settings.py`에 추가:

```python
CSRF_COOKIE_HTTPONLY = False        # JS가 쿠키를 읽어 헤더에 실어야 하므로 False (기본값)
CSRF_TRUSTED_ORIGINS = ["http://localhost", "http://localhost:5173"]
SESSION_COOKIE_SAMESITE = "Lax"
```

CSRF 쿠키 발급 엔드포인트 (프론트가 앱 시작 시 1회 호출):

```python
from django.views.decorators.csrf import ensure_csrf_cookie
from django.http import JsonResponse

@ensure_csrf_cookie
def csrf(request):
    return JsonResponse({"detail": "CSRF cookie set"})
```

로그인 FBV는 Django Form으로 검증한다 — 평가 항목 "폼 검증"의 실증 코드가 된다.

```python
from django import forms
from django.contrib.auth import authenticate, login

class LoginForm(forms.Form):
    email = forms.EmailField()
    password = forms.CharField(min_length=8)

def login_view(request):
    form = LoginForm(json.loads(request.body))
    if not form.is_valid():
        return JsonResponse({"message": "입력값이 올바르지 않습니다.",
                             "detail": form.errors}, status=400)
    user = authenticate(request, username=form.cleaned_data["email"],
                        password=form.cleaned_data["password"])
    if user is None:
        return JsonResponse({"message": "이메일 또는 비밀번호가 일치하지 않습니다."}, status=401)
    login(request, user)
    return JsonResponse({"id": user.id, "email": user.email})
```

에러 응답은 항상 `{"message": ..., "detail": ...}` JSON으로 반환한다. Django 기본 에러 페이지(HTML)가 나가면 프론트 `ApiError` 파싱이 실패한다.

### 6.2 프론트 client.ts 수정 (CSRF + 타임아웃)

`frontend/src/api/client.ts`의 `request` 함수에 두 가지를 추가한다. 현재 코드에는 CSRF 헤더와 타임아웃이 둘 다 없다 — **인증 연결 첫날 이걸 안 하면 403으로 하루를 버린다.**

```typescript
/** Django가 내려준 csrftoken 쿠키를 읽는다 */
function getCookie(name: string): string | null {
  const match = document.cookie.match(new RegExp(`(?:^|; )${name}=([^;]*)`));
  return match ? decodeURIComponent(match[1]) : null;
}

// request() 내부 fetch 호출을 아래처럼 수정
const controller = new AbortController();
const timeoutMs = baseUrl === NAMING_API_BASE ? 90_000 : 15_000; // 작명 90초, 일반 15초
const timer = setTimeout(() => controller.abort(), timeoutMs);

const csrfToken = getCookie("csrftoken");
res = await fetch(`${baseUrl}${path}`, {
  ...rest,
  credentials: "include",
  signal: controller.signal,
  headers: {
    "Content-Type": "application/json",
    Accept: "application/json",
    ...(csrfToken ? { "X-CSRFToken": csrfToken } : {}),
    ...headers,
  },
  body: body === undefined ? undefined : JSON.stringify(body),
});
clearTimeout(timer);
```

`catch` 블록에서 `AbortError`를 구분해 "요청 시간이 초과되었습니다" 메시지를 던지도록 한다 (`cause instanceof DOMException && cause.name === "AbortError"`).

### 6.3 검증

vite dev 서버(5173)에서 로그인을 시도한다. 개발자도구 Network 탭에서 확인할 것: ① `POST /api/auth/login` 요청 헤더에 `X-CSRFToken`이 있다, ② 응답에 `sessionid` Set-Cookie가 있다, ③ 이후 `GET /api/me` 류 요청이 200을 받는다. 세 가지가 모두 확인되면 인증 연결 완료다.

## 7. STEP 5 — USE_MOCK 도메인별 분리와 점진 전환 【2차】

현재 `USE_MOCK`은 전역 스위치라 false로 바꾸는 순간 13개 API가 전부 실서버로 향한다. 인증만 먼저 붙이고 나머지는 mock을 유지할 수 있도록 도메인별 플래그로 나눈다.

`client.ts`:

## 9. 예상 오류와 해결 방법 (트러블슈팅)

작업 단계별로 만날 가능성이 높은 순서로 정리했다. 증상 → 원인 → 해결 순이다.

### A. 폴더 병합 단계

**A-1. `npm run dev`에서 `Cannot find module '@/...'` 또는 import 에러**
원인: 이동 후 `node_modules` 없이 실행했거나, 에디터가 옛 demo3 경로의 tsconfig를 참조.
해결: `frontend/`에서 `npm ci` 재실행. VS Code는 `SKN29-4th-4Team`을 루트로 다시 연다. `vite.config.ts`의 `@` alias는 `__dirname` 기준 상대 경로라 이동해도 깨지지 않는다.

**A-2. `git add .`이 수만 개 파일을 스테이징하거나 멈춤**
원인: `.gitignore`에 `frontend/node_modules/` 추가 전에 add 실행.
해결: `git reset` 후 3.3의 .gitignore를 먼저 커밋하고 다시 add. 이미 커밋했다면 `git rm -r --cached frontend/node_modules`.

**A-3. Windows에서 CRLF 경고가 대량 발생**
원인: 두 레포의 줄바꿈 설정(autocrlf)이 다름.
해결: 레포 루트에 `.gitattributes` 추가 — `* text=auto eol=lf`. 경고 자체는 무해하므로 급하면 무시해도 된다.

**A-4. 한글 파일명이 깃에서 `"\355\224\204..."`로 깨져 보임**
원인: 깃의 경로 인용 설정.
해결: `git config core.quotepath false`. 표시 문제일 뿐 파일은 정상이다.

### B. 도커 빌드 단계

**B-1. `COPY frontend/package*.json: not found` (가장 먼저 밟는 오류)**
원인: compose의 nginx `build.context`가 여전히 `./deploy/nginx`라 `frontend/`가 컨텍스트 밖.
해결: 4.2대로 `context: .` + `dockerfile: deploy/nginx/Dockerfile`로 수정.

**B-2. `npm ci` 실패 — `lockfile version mismatch` 또는 `EBADPLATFORM`**
원인: 로컬 node 버전과 이미지(node:20)의 lockfile 해석 차이, 또는 Windows에서 만든 lockfile의 optional deps 문제(rollup 네이티브 바이너리).
해결: `frontend/`에서 로컬 node 20으로 `rm package-lock.json node_modules; npm install`(PowerShell은 `Remove-Item -Recurse -Force`) 후 재커밋. 그래도 rollup 에러가 나면 Dockerfile의 `npm ci`를 `npm ci --no-optional && npm i @rollup/rollup-linux-x64-musl`로 임시 대체하거나 `node:20-slim`(alpine 아님)으로 변경.

**B-3. 빌드는 성공했는데 `http://localhost`가 nginx 기본 페이지("Welcome to nginx")**
원인: `COPY --from=build /app/dist`가 빈 폴더거나 nginx.conf의 root 경로 불일치.
해결: `docker compose build --no-cache nginx` 후 `docker run --rm <이미지> ls /usr/share/nginx/html`로 index.html 존재 확인. `vite build` 출력 경로(dist)와 COPY 경로가 일치하는지 확인.

**B-4. 빌드 컨텍스트 전송이 수 분 걸림**
원인: `.dockerignore` 누락으로 `node_modules`·`data/`까지 전송.
해결: 3.3의 `.dockerignore` 추가.

### C. nginx·라우팅 단계

**C-1. 로그인 요청이 404 — 응답 헤더가 FastAPI(uvicorn)**
원인: 옛 nginx.conf가 `/api/`를 fastapi:8001로 보내는 채로 남음.
해결: 4.3의 conf로 교체 후 `docker compose restart nginx`. conf가 이미지에 구워지므로 restart가 아니라 **rebuild**가 필요할 수 있다.

**C-2. SPA에서 새로고침하면 404 (예: /results에서 F5)**
원인: `try_files $uri /index.html;` 누락.
해결: 4.3의 `location /` 블록 확인.

**C-3. 작명 생성이 정확히 60초 뒤 504**
원인: nginx 기본 `proxy_read_timeout`(60s)이 LLM 응답보다 짧음.
해결: `/naming-api/` 블록에 `proxy_read_timeout 120s`. 프론트 AbortController 타임아웃(90s)은 이보다 짧게 유지해 사용자에게 먼저 안내한다.

**C-4. `/naming-api/names/generate`가 404인데 FastAPI 직접 호출(8001)은 200**
원인: `proxy_pass` 끝 슬래시로 인한 접두사 처리 불일치 (4.3 참고).
해결: FastAPI 라우터 경로가 `/names/generate`면 `proxy_pass http://fastapi:8001/;`(슬래시 O), `/naming-api/names/generate`면 슬래시 X. `docker compose logs fastapi`로 실제 수신 경로를 확인하면 즉시 판별된다.

**C-5. Django Admin이 CSS 없이 깨진 화면**
원인: collectstatic 미실행 또는 `/static/` 볼륨 미연결.
해결: django 컨테이너에서 `python manage.py collectstatic --noinput` 실행, compose에서 staticfiles 볼륨을 nginx와 공유하고 `location /static/`의 alias 경로 일치 확인.

### D. 인증·API 연결 단계

**D-1. 모든 POST가 403 `CSRF verification failed`**
원인: `X-CSRFToken` 헤더 누락(현재 client.ts에 없음) 또는 csrftoken 쿠키 자체가 없음.
해결: 6.2의 client.ts 수정 + 앱 시작 시 `GET /api/auth/csrf` 1회 호출. vite(5173)에서 나면 `CSRF_TRUSTED_ORIGINS`에 `http://localhost:5173` 추가 확인.

**D-2. 로그인은 성공하는데 다음 요청부터 401**
원인: sessionid 쿠키가 저장·전송되지 않음.
해결: ① `credentials: "include"` 확인(이미 구현됨), ② **반드시 vite proxy 경유로 호출** — 팀원이 `http://localhost:8000`을 직접 치면 cross-origin이라 쿠키가 막힌다. "API는 항상 상대경로(/api)로 호출"을 팀 규칙으로 공지, ③ `SESSION_COOKIE_SECURE=True`가 로컬(http)에 설정돼 있으면 False로.

**D-3. 프론트 에러 화면에 "API 요청 실패: 500"만 뜨고 원인 불명**
원인: Django가 JSON이 아니라 HTML 에러 페이지를 반환해 `detail` 파싱 실패.
해결: 모든 FBV에서 예외를 잡아 `{"message", "detail"}` JSON으로 반환. 공통 미들웨어나 데코레이터로 처리하면 누락이 없다. 디버깅은 `docker compose logs django`.

**D-4. 작명 결과 카드가 빈 값/undefined로 렌더링**
원인: FastAPI 응답 필드가 snake_case(`last_name`)이거나 필드 누락 — 프론트는 camelCase를 기대.
해결: 5.2처럼 Pydantic 모델 필드명을 types.ts와 글자 단위로 일치시킨다. `response_model=list[NameResult]`를 지정하면 Pydantic이 누락 필드를 빌드 타임에 잡아준다.

**D-5. 422 Unprocessable Entity (FastAPI)**
원인: `NameRequest`가 union 타입(`natural`/`structured`)인데 한쪽 스키마로만 받도록 정의.
해결: `type` 필드로 분기하는 discriminated union(Pydantic `Field(discriminator="type")`) 또는 dict로 받아 내부 분기.

### E. LLM·데이터 단계

**E-1. Structured Output이 간헐적으로 스키마를 어김**
원인: 프롬프트 기반 JSON 요청은 비결정적.
해결: `with_structured_output()` API를 사용해 모델 레벨에서 강제. 그래도 실패하면 1회 재시도 후 502 `{"message": "작명 생성에 실패했습니다"}` 반환. 이 실패 케이스가 테스트 보고서의 "LLM 실패 검증" 항목이 된다.

**E-2. Neo4j 컨테이너가 기동 몇 분 후 죽음 (exit 137)**
원인: 메모리 부족(OOM Kill) — Java 힙 기본값이 호스트 메모리를 초과.
해결: 4.2의 heap/pagecache 캡 추가. 로컬 8GB 미만 팀원은 compose 프로파일에서 neo4j 제외하고 개발.

**E-3. LLM 계약 테스트를 어떻게 쓰나**
방법: 답변 내용이 아니라 **계약**을 테스트한다. `fastapi_app/main.py`의 전역 `_graph`를 pytest monkeypatch로 스텁 교체 후 TestClient로 ① 응답이 NameResult 스키마 준수, ② 실패 시 에러 형식, ③ 타임아웃 시 504를 검증. 반나절 작업으로 평가표 테스트 항목 3개를 커버한다.

### F. 환경·기타

**F-1. `VITE_USE_MOCK=false`로 바꿨는데 여전히 mock 동작**
원인: VITE_ 환경변수는 **빌드/기동 시점**에 주입된다. dev 서버 재시작 또는 도커 재빌드 전에는 반영 안 됨.
해결: dev는 `npm run dev` 재시작, 도커는 `docker compose build nginx` 재실행. 또한 `.env`가 `frontend/` 안에 있는지 확인 — 레포 루트 `.env`는 vite가 읽지 않는다.

**F-2. 포트 충돌 — `bind: address already in use`**
원인: 80/8000/8001/5173 중 다른 프로세스가 점유.
해결: `netstat -ano | findstr :80`으로 PID 확인 후 종료, 또는 compose 포트 매핑 변경.

**F-3. 환경변수 파일을 합쳐버림**
규칙: `frontend/.env`(VITE_*)와 루트 `.env`(Django/FastAPI/DB 시크릿)는 **절대 합치지 않는다.** VITE_ 변수는 빌드 시 번들에 그대로 노출되므로, 백엔드 시크릿을 같은 파일에 두면 유출 사고 경로가 된다.

**F-4. CORS 에러가 보인다면**
현 구성(vite proxy + nginx 단일 도메인)에서는 CORS가 발생할 수 없다. CORS 에러가 보인다면 누군가 프록시를 우회해 `http://localhost:8001` 등을 직접 호출한 것이다. `django-cors-headers` 설치가 아니라 **호출 경로를 상대경로로 고치는 것**이 해결책이다.

- [ ] 회원가입 → 로그인 → 새로고침 후에도 로그인 유지 (세션)
- [ ] 작명 생성 → 결과 카드에 한자·오행·수리 정상 표시 (실 FastAPI 응답)
- [ ] 마이페이지 이력에 방금 생성한 기록 표시 (History DB)
- [ ] `/admin/`에서 User·History CRUD 동작
- [ ] SPA 임의 경로(예: /results)에서 F5 → 404 없이 화면 유지
- [ ] 백엔드 중지 상태에서 작명 시도 → 90초 내 타임아웃 에러 UI 표시
- [ ] `docker compose down && up` 재기동 후 데이터 유지 (볼륨)
- [ ] 산출물 갱신: 화면설계서에 관리자 화면 "Django Admin 이관" 반영, 시스템 구성도에 React 빌드 흐름 추가, 테스트 보고서에 LLM 성공/실패/타임아웃 + 컨테이너 재현성 포함
